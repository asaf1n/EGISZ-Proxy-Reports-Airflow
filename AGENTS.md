# AGENTS Instructions

Этот файл задает правила работы LLM-агентов в репозитории `EGISZ-Proxy-Reports-Airflow`.

## Цель агента

- Поддерживать рабочий ETL-пайплайн Firebird -> PostgreSQL.
- Не ломать сценарий локального запуска через `docker compose`.
- Сохранять совместимость с Airflow DAG и автопровижинингом Metabase.

## Границы изменений

- Вносить минимально необходимые правки.
- Не менять имена сервисов в `docker-compose.yml` без явного запроса.
- Не удалять существующие env-переменные и их fallback-значения без причины.
- Не внедрять новую бизнес-логику в DAG/ETL, если задача про инфраструктуру или документацию.

## Обязательные проверки после правок

Если затронут ETL/Python-код:

```powershell
py -m build
```

Если затронут Docker/Metabase/Airflow/скрипты запуска:

```powershell
docker compose build
```

Если затронуто сразу несколько подсистем — выполнить обе проверки.

## Работа с Docker и Metabase

- Учитывать, что `metabase/Dockerfile` должен собираться полностью (включая вспомогательные shell-скрипты).
- Любой новый скрипт в `metabase/` делать исполняемым в Linux-контейнере (через `chmod +x` в Dockerfile или существующий RUN-блок).
- Не ломать текущий процесс провижининга: `entrypoint.sh` -> `provision.sh` -> `setup-dashboards.sh`.

## Работа с ETL

- Предпочитать идемпотентные изменения.
- Сохранять поддержку запуска CLI-команд через entrypoint `python -m proxy_reports_etl.cli`.
- Не менять ожидаемые env-ключи (`FB_*`, `PG_*`, `ETL_*`) без миграционного плана и обновления README.

## Документация

- При изменении команд запуска/сборки обновлять `README.md` в том же PR/коммите.
- Инструкции в README писать для пользователя, без внутренних рассуждений агента.
- Если изменилось поведение сборки модулей, явно отражать это в разделе «Сборка всех модулей».

## Стиль изменений

- Маленькие, проверяемые коммиты.
- Никаких массовых рефакторингов без запроса.
- Новые файлы и команды должны быть воспроизводимы в текущем окружении Windows + Docker.

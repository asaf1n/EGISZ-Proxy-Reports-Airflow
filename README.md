# Сервис эксплуатационной аналитики обмена с ЕГИСЗ

Витрина аналитики **«Интеграция с ЕГИСЗ»**: сервис инкрементально вычитывает журнал обмена проксирующего шлюза из Firebird (`proxy_egisz`), парсит SOAP/XML на стороне PostgreSQL, склеивает все сообщения одного документа в единую запись, нормализует статусы и причины отказов и формирует фактовую витрину уровня документа, поверх которой строятся дашборды Metabase.

Сервис отвечает на эксплуатационные вопросы: какие клиники отправляют документы, дошёл ли по каждому СЭМД ответ от РЭМД, какие документы отвергнуты и почему, нет ли пропусков в обработке и жив ли сам ELT-контур.

## Контекст: ЕГИСЗ, СЭМД, обмен и где в нём наш контур

**ЕГИСЗ** — единая государственная информационная система в сфере здравоохранения. С точки зрения медицинской организации (МО) ключевое взаимодействие с ЕГИСЗ — это передача структурированных электронных медицинских документов (**СЭМД**) в подсистему **РЭМД** (реестр электронных медицинских документов). Каждый СЭМД — выписной эпикриз, протокол осмотра, направление, заключение и десятки других видов — должен быть зарегистрирован в РЭМД.

Обмен с РЭМД асинхронный, по схеме «отправил — получил подтверждение позже». МИС клиники формирует CDA-документ, оборачивает его в SOAP-конверт, ставит ЭП, отправляет в РЭМД и получает синхронный технический ответ («принято к обработке»). Спустя секунды или часы приходит асинхронный callback с результатом: документ либо зарегистрирован и получил `emdrId`, либо отклонён со списком ошибок (Schematron, XSD, бизнес-правила, проблемы с подписью или справочниками НСИ). Одна логическая «отправка СЭМД» — это, как правило, **минимум два сообщения** в журнале обмена: исходящее и его callback.

Наша компания работает как **интегратор**: между МИС клиник и инфраструктурой ЕГИСЗ стоит проксирующий шлюз. Он принимает от МИС уже подписанный документ, маршрутизирует его в РЭМД, получает callback и возвращает его клинике. Все эти сообщения шлюз пишет в свою журнальную БД — **`proxy_egisz`** на Firebird 5. Эта БД — нормализованный лог обмена для нужд самого шлюза, она не приспособлена ни для аналитики, ни для эксплуатационных отчётов.

Сервис работает поверх этого журнала: инкрементально вычитывает новые строки, парсит SOAP/XML на стороне PostgreSQL, собирает из разрозненных сообщений единую запись документа, нормализует статусы и причины отказов и формирует витрину, на которой строятся дашборды Metabase.

## Что делает сервис и на какие вопросы отвечает

С практической точки зрения сервис закрывает пять задач.

**Контроль отправки.** Объём отправки СЭМД через шлюз: число документов в час и в день, разрезы по клинике и типу документа, провалы и всплески. Прекращение отправки клиникой, как правило, указывает на сбой в её МИС и проявляется на графиках раньше, чем поступает обращение в поддержку.

**Контроль обратной связи.** Видеть, что на каждый исходящий СЭМД пришёл callback от РЭМД и что callback корректно склеился с оригиналом по идентификаторам (`messageId`, `relatesTo`, `localUid`, `emdrId`). Документ, по которому callback не пришёл за разумное окно (обычно более 24 ч), — кандидат на эскалацию.

**Контроль ошибок.** Видеть, какие СЭМД отвергнуты, и понимать причину в человекочитаемом виде. РЭМД возвращает ошибки длинными XML-конвертами с десятками `<item>` внутри, часть из них не имеет внятного человеческого текста (только код `RUNTIME_ERROR` и т. п.). Сервис парсит этот XML и приводит каждую ошибку к одной из канонических категорий (СНИЛС пациента, специальность врача не в НСИ, истёк сертификат организации, сетевой таймаут, дубликат и так далее).

**Контроль полноты.** Видеть, нет ли пропусков в обработке: документы без `emdrId`, документы со «странным» статусом, документы без ответа, пустые поля там, где их быть не должно.

**Контроль самого сервиса.** Видеть, что ELT-контур жив: курсор двигается, материализованные представления свежие, отставание DAG в пределах нормы, нет накопившихся ошибок ETL. Под это есть отдельный дашборд (`02_service.json`).

## Архитектура и поток данных

Сервис построен по схеме **ELT**: извлечение и загрузка — на Python внутри Airflow, трансформация — на стороне PostgreSQL в функциях PL/pgSQL. Предметная логика (парсинг XML, нормализация статусов, сборка документа, классификация ошибок, формирование фактовой таблицы) сосредоточена в SQL-слое DWH (`db/parts/`) и может выполняться и тестироваться непосредственно в базе, без запуска Airflow. Python-слой выполняет только оркестрацию и загрузку и не содержит предметной логики.

| Компонент    | База / технология                          | Что делает                                                                                                   |
| ------------ | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| Источник     | Firebird 5, БД `proxy_egisz`               | Журнал обмена шлюза. Не правим, только читаем.                                                               |
| Оркестратор  | Apache Airflow 2.11.2, DAG `egisz_elt_dag` | Каждые 5 минут забирает дельту журнала, грузит в raw, парсит и склеивает документы, обновляет mat-view'ы, двигает курсор. |
| DWH          | PostgreSQL, БД `dwh_egisz`                  | Raw-слой, справочники, разбор сообщений, фактовая таблица документов, функции классификации, материализованные и обычные view. |
| Визуализация | Metabase v0.60.2.5                         | 8 дашбордов поверх `v_egisz_documents_*_ui` и `v_rpt_*_ui`.                                                  |
| Деплой       | Kubernetes (Docker Desktop по умолчанию)   | Helm-чарт Airflow + ручной манифест Metabase, общий скрипт `up.ps1`.                                         |

Сетевые порты для локального dev-стенда: Airflow на `localhost:8080`, Metabase на `localhost:3000`.

Служебные базы разделены: Airflow держит метабазу `airflow_db` (внутри Helm-чарта, в Kubernetes PVC), Metabase — служебную `metabase_app`. Аналитический DWH `dwh_egisz` вынесен отдельно и не используется для служебных нужд приложений.

Подключения хранятся в Airflow Connections, имена фиксированы: `proxy_egisz_fb` для Firebird-источника и `dwh_egisz_pg` для DWH. В DAG используются только Airflow Connections (`BaseHook.get_connection`), без `os.getenv` или `.env`. Это позволяет переподключать инстансы (например, переключить чтение на read-replica) без правок кода.

## Источник: прокси-БД интегратора (Firebird `proxy_egisz`)

Из десятков таблиц шлюза сервис читает три. Основной отбор инкрементальный по монотонно растущему ключу `EXCHANGELOG.LOGID`: Firebird поддерживает keyset-пагинацию через `WHERE LOGID > :last ORDER BY LOGID ROWS :limit`, а `LIMIT/OFFSET` на этом диалекте использовать нельзя.

| Таблица источника | Что это                | Ключевые поля                                                        | Как используется                                                                                  |
| ----------------- | ---------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `EXCHANGELOG`     | Основной журнал обмена | `LOGID` (PK, монотонный bigint), `LOGDATE`, `CREATEDATE`, `MSGID`, `LOGSTATE`, `LOGTEXT`, `MSGTEXT` | Каждая строка — одно сообщение между шлюзом и РЭМД (исходящее или callback). `MSGTEXT` (полный SOAP/XML payload) — основной материал для парсинга. |
| `JPERSONS`        | Справочник организаций | `JID`, `JNAME`, `JINN`, `JADDR`                                     | Грузится в `dim_organizations` (`jid`, `name`, `inn`, `address`). Даёт человекочитаемое имя клиники по `JID`. |
| `EGISZ_LICENSES`  | Справочник лицензий    | `mo_uid`, `JID`                                                     | Грузится в `dim_licenses`. Связывает OID организации (`mo_uid`) с внутренним `JID` для обогащения callback'ов. |

`MSGTEXT` хранит полный SOAP/XML payload как BLOB: из него извлекаются идентификаторы документа, код и название СЭМД, статус и ошибки. `EXCHANGELOG.LOGDATE` — служебная дата журнала и для аналитики документов не используется.

## ELT-конвейер Airflow

DAG `egisz_elt_dag` собран на TaskFlow API, запускается каждые 5 минут (`*/5 * * * *`), `max_active_runs=1` — параллельные прогоны запрещены, что исключает гонку за watermark. Метаданные батча передаются между задачами через XCom. Размер батча задан константой `BATCH_SIZE = 5000`.

DWH-схема создаётся **не задачей DAG**, а отдельным прогоном `db/dwh_init.sql` (см. [DWH-модель](#dwh-модель)).

Последовательность задач:

```
sync_dimensions
  ▸ extract_and_load_batch
    ▸ analyze_staging
      ▸ transform_data
        ▸ refresh_materialized_views
          ▸ update_watermark
            ▸ reconcile_late_arrivals
```

| Задача                       | Что делает                                                                                                                                            | Идемпотентность / гарантии                                                                          |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `sync_dimensions`            | Полная синхронизация справочников `dim_organizations` (из `JPERSONS`) и `dim_licenses` (из `EGISZ_LICENSES`). Стоит первой, чтобы остальные шаги работали с актуальными справочниками. | UPSERT через `ON CONFLICT`; новые клиники и лицензии видны сразу в первом же прогоне.                |
| `extract_and_load_batch`     | Читает курсор `last_logid` из `elt_state`, берёт батч `EXCHANGELOG` по `LOGID > last_logid` и в том же шаге грузит сырые строки в `exchangelog_raw`.  | UPSERT `INSERT ... ON CONFLICT DO UPDATE` (PK `logid`): повторный батч не даёт дублей.               |
| `analyze_staging`            | `ANALYZE` raw-таблиц, в которые в этом батче что-то загружалось.                                                                                      | Освежает planner-статистику после bulk-загрузки; обязательный шаг перед трансформацией.              |
| `transform_data`            | Парсит сырой `MSGTEXT` в `dim_egisz_exchangelog_refs`, собирает документы и обновляет `fact_egisz_documents`. Затем `ANALYZE` фактовых таблиц.        | Парсинг каждой строки журнала ровно один раз; UPSERT по `document_key` собирает все сообщения документа в одну запись. |
| `refresh_materialized_views` | Обновляет материализованные витрины через `REFRESH MATERIALIZED VIEW CONCURRENTLY`, затем `ANALYZE` обновлённых mat-view.                            | `CONCURRENTLY` не блокирует SELECT'ы от Metabase (требует уникального индекса на mat-view).          |
| `update_watermark`           | UPSERT курсора `last_logid` в `elt_state`.                                                                                                            | Сдвиг через `GREATEST(current, new)`: сбой порядка прогонов или ручной запуск старого батча не откатят курсор назад. |
| `reconcile_late_arrivals`    | Догружает строки `EXCHANGELOG`, которые материализовались в источнике уже **ниже** текущего watermark (вне порядка `LOGID`), сверяя полосу под курсором с `exchangelog_raw`. | Закрывает «дыры» от опоздавших строк без отката курсора; повторный прогон безопасен.                 |

Watermark двигается только после успешной трансформации и обновления витрин: при падении любого шага те же данные перечитываются на следующем прогоне.

## Парсинг и нормализация payload

Ключевой принцип: **`MSGTEXT` хранится как сырой payload, а бизнес-поля извлекаются в DWH SQL-функциями**. Это оставляет исходный журнал неизменным и позволяет пересчитать витрину при изменении правил парсинга. Python не парсит SOAP/XML: DAG остаётся тонким, а правила парсинга проверяются прямыми SELECT-запросами в базе.

Утилитные функции парсинга и нормализации (`db/parts/20_functions_parsing.sql`):

| Функция                              | Назначение                                                                                                       |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| `egisz_xml_text(payload, tag_name)`  | Извлекает текст между тегами `<*:tag_name>...` агностично к namespace-префиксу; безопасно экранирует имя тега; при ошибке возвращает `NULL`, чтобы одна битая строка не сломала весь батч. |
| `egisz_normalize_message_id(value)`  | Приводит идентификаторы вида `<urn:uuid:...>`, `urn:uuid:...`, голый UUID к одной канонической форме. На неё повешены функциональные индексы для быстрых JOIN-ов по идентификаторам. |
| `safe_cast_timestamptz(text)`        | Безопасный каст текста в `timestamptz` (возвращает `NULL` вместо исключения) — даты в payload приходят в разных форматах. |
| `egisz_clean_host` / `egisz_extract_jid_from_endpoint` | Разбирают эндпоинт callback вида `gost-<JID>....` — вытаскивают `JID` клиники и нормализуют хост для JOIN со справочником. |
| `egisz_clean_text_value(text)`       | Общая нормализация текстового поля (схлопывание whitespace, удаление BOM/неразрывных пробелов), чтобы в UI не появлялись «дубликаты» из-за пробельного шума. |
| `egisz_normalize_semd_code(text)`    | Нормализация кода СЭМД (приходит с разными префиксами/суффиксами шаблонов).                                       |

Поля, которые извлекаются из payload и попадают в разбор сообщения (`dim_egisz_exchangelog_refs`) и далее в документ:

| Поле                | Источник                                          | Правило                                                                          |
| ------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------- |
| MSGID обмена        | `EXCHANGELOG.MSGID` или XML `<messageId>`         | Приоритет XML `<messageId>`, fallback — `MSGID` из журнала.                       |
| Связанное сообщение | XML `<relatesToMessage>` или `<relatesTo>`        | Коррелятор исходящего запроса и callback.                                         |
| Локальный ID СЭМД   | XML `<localUid>` или `<DOCUMENTID>`               | Приоритет `localUid`, fallback — `DOCUMENTID`.                                    |
| Федеральный ID      | XML `<emdrId>`                                    | Идентификатор документа на стороне РЭМД, если есть в payload.                     |
| Номер документа     | XML `<documentNumber>`                            | Дополнительный fallback для ключа учёта.                                          |
| OID организации     | XML `<organization>`                              | Связь с лицензиями через `dim_licenses.mo_uid`.                                   |
| Код СЭМД            | XML `<kind>` / `<KIND>`                            | Поддерживаются оба регистра тега.                                                 |
| Название СЭМД       | XML `<name>` / `<documentName>`                   | Человекочитаемая подпись типа документа.                                          |
| Код / текст ошибки  | XML `<code>` / `<message>`, для `LOGSTATE = 3` — `LOGTEXT` | Вход для классификации ошибок.                                            |
| Дата создания СЭМД  | XML `<creationDateTime>` / `<creationDate>`       | Приводится к `timestamptz` безопасным cast.                                       |
| JID клиники         | эндпоинт callback, pattern `gost-<JID>`           | Поздняя подсказка: один хост может обслуживать несколько клиник.                  |

**Нормализация статуса** вычисляется в SQL: `LOGSTATE = 3` всегда трактуется как `error` (транспортная ошибка канала); XML-статус `success` даёт `success`; XML-статус `error` или payload, содержащий `error`, даёт `error`; всё остальное — `unknown`.

**Код и наименование СЭМД.** Код СЭМД — это OID из справочника НСИ `1.2.643.5.1.13.13.11.1520`. Наименование берётся из `dim_semd_types`, либо payload fallback, если код ещё не заведён в справочнике. Справочник seedится в `db/parts/10_tables.sql`; `code` соответствует OID справочника, `type_code` хранится отдельно. Обновление справочника — заменить seed-данные и переимпортировать Metabase.

## Сборка документа из сообщений

Одна логическая отправка СЭМД распределена по нескольким строкам `EXCHANGELOG` (исходящее сообщение, технический ответ, поздний callback). Сервис сводит их в одну запись документа в два слоя:

1. **Разбор сообщения** — `transform_data` парсит каждую новую строку `exchangelog_raw` ровно один раз и складывает разобранные поля в `dim_egisz_exchangelog_refs` (PK `logid`). Дальше трансформация читает только этот разобранный слой, а не сырой XML.

2. **Ключ документа.** Все сообщения, относящиеся к одному СЭМД, сводятся под общий `document_key`, который вычисляется по приоритету идентификаторов: `localUid` / `DOCUMENTID` → `emdrId` → `relatesToMessage` → номер документа → `messageId`. Идентификаторы предварительно нормализуются (`egisz_normalize_message_id`), поэтому callback склеивается с исходным сообщением независимо от формы записи UUID.

3. **Запись документа** — UPSERT в `fact_egisz_documents` (PK `document_key`). Одна строка = один документ. Она накапливает временны́е метки (`first_sent_at`, `last_callback_at`, `registered_at`), финальный статус (`last_status`, `status_category`), идентификаторы (`local_uid`, `emdr_id`, `message_id`, `relates_to_id`), привязку к клинике (`jid`) и поля ошибок.

Опоздавшие callback'и, материализовавшиеся в источнике ниже watermark, добираются задачей `reconcile_late_arrivals` и обновляют уже существующую запись документа — без отката курсора.

## Классификация ошибок

РЭМД возвращает ошибки несколькими способами одновременно: машиночитаемым кодом в SOAP-faultcode (`VALIDATION_ERROR`, `RUNTIME_ERROR`, `ASYNC_RESPONSE_TIMEOUT` и др.), плюс текстом, плюс отдельными `<item>` внутри ответного XML (часто десяток в одном callback), плюс инлайн-кусками Schematron в человеческом тексте. Если складывать сырой текст как есть — получатся тысячи уникальных формулировок, которые невозможно агрегировать в дашборде.

Поэтому каждая ошибка проходит через классификатор и приводится к одной из канонических категорий. Классификация декларативная: правила лежат в таблице `egisz_error_interpretation_rules` (`db/parts/30_error_rules.sql`) и применяются функциями из `db/parts/40_functions_errors.sql` по алгоритму:

1. Перебрать активные правила по возрастанию `priority`. Выигрывает первое, у которого совпали и `match_code` (если задан), и `match_pattern` (regex по тексту сообщения).
2. Если ни одно правило не сработало — применяются захардкоженные code-fallback'и (`RUNTIME_ERROR`/`INTERNAL_ERROR`, `CA_INACCESSIBILITY`/`CA_UNAVAILABLE`, `ASYNC_RESPONSE_TIMEOUT`/`TIMEOUT`).
3. Если не сработали и они — возвращается `Неизвестная ошибка`.

Внутри одного callback РЭМД может прислать несколько `<item>` разных типов: они дедуплицируются и склеиваются через `·`. Случаи `Неизвестная ошибка` видны в дашборде `04_quality_and_errors.json` и служат сигналом, что в правила нужно добавить новую формулировку.

Канонический результат сохраняется в фактовой таблице в трёх полях:

| Поле           | Содержание                                                                       |
| -------------- | -------------------------------------------------------------------------------- |
| `error_type`   | Плоская канонизированная категория для группировки в дашбордах. Несколько типов склеиваются через `·`. |
| `error_summary`| Человекочитаемая сводка по правилам (для карточек).                              |
| `error_text`   | Исходные тексты `<message>` без классификации (для drill-down).                  |

View `v_rpt_error_interpretations_ui` раскрывает ошибки построчно (одна строка на `<item>`) — для drill-down и аудита правил.

Правила сгруппированы по содержательным категориям (поле `error_category` в seed); ниже — фактический состав:

| Группа | Правил |
| --- | --- |
| Ошибки регистрации в РЭМД | 19 |
| Данные пациента | 15 |
| Данные медработника | 10 |
| Ошибки структуры и валидации | 9 |
| Ошибки ЭП и сертификатов | 9 |
| Ошибки организации / ИС | 7 |
| Ошибки получения файла ЭМД | 6 |
| Технические ошибки РЭМД | 6 |
| Ошибки справочника НСИ | 4 |
| Ошибки связи | 1 |

Полная таблица правил (с конкретными `code` и regex) — в [приложении ниже](#приложение-полный-список-правил-классификации); источник истины — `db/parts/30_error_rules.sql`.

## DWH-модель

Схема создаётся скриптом `db/dwh_init.sql`, который через `\i` подключает 11 модулей из `db/parts/` в порядке зависимостей. Отдельных миграционных файлов нет: чистый dev-DWH разворачивается одним прогоном, изменения схемы вносятся в соответствующий модуль через идемпотентные конструкции (`CREATE TABLE IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, `ALTER TABLE ... ADD/DROP COLUMN IF EXISTS`, `INSERT ... ON CONFLICT`). Скрипт идемпотентен: повторный запуск не пересоздаёт и не теряет данные. Перед выполнением он проверяет, что подключён к `dwh_egisz`.

```
psql -U postgres -d dwh_egisz -v ON_ERROR_STOP=1 -f db/dwh_init.sql
```

Слои DWH:

| Слой        | Объект                          | Назначение                                                                                  |
| ----------- | ------------------------------- | ------------------------------------------------------------------------------------------- |
| Состояние   | `elt_state` (PK `pipeline`)     | Курсор инкрементальной обработки `last_logid`.                                              |
| Raw         | `exchangelog_raw` (PK `logid`)  | Строки `EXCHANGELOG` в исходном виде, без трансформации; `MSGTEXT` хранится как `text`.     |
| Разбор      | `dim_egisz_exchangelog_refs` (PK `logid`) | Разобранные поля каждой строки журнала (идентификаторы, статус, ошибки, `document_key`); парсятся один раз, дальше читаются трансформацией. |
| Справочник  | `dim_organizations` (PK `jid`)  | Имя клиники, ИНН, адрес из `JPERSONS`.                                                       |
| Справочник  | `dim_licenses` (PK `id`)        | Привязка `mo_uid` ↔ `JID` для обогащения callback'ов.                                       |

## Аналитика и дашборды Metabase

Восемь дашбордов покрывают разные роли — от дежурной поддержки до руководства и клиента. Каждый отвечает на свой вопрос:

| Дашборд (файл)                          | Кому                        | На какой вопрос отвечает                                                              |
| --------------------------------------- | --------------------------- | ------------------------------------------------------------------------------------- |
| Оперативный мониторинг (`01_operational.json`) | Дежурная поддержка     | Что происходит с потоком отправки сейчас — объёмы за час и день, динамика доли ошибок, преобладающие типы ошибок. |
| Сервис и healthcheck (`02_service.json`) | Поддержка ETL              | Жив ли контур — отставание загрузки, доля сбоев канала, объёмы по клиникам.            |
| Документы без ответа (`03_documents_no_response.json`) | Поддержка        | По каким СЭМД не пришёл callback за норматив (>24 ч) — рабочая очередь эскалации.      |
| Ошибки и качество данных (`04_quality_and_errors.json`) | Аналитик / поддержка | Почему документы отклоняются — успешность, структура причин отказов, полнота данных.   |
| Сводная статистика (`05_executive.json`)| Руководство               | Как обстоят дела в целом — общая доля успеха, ведущие причины отказов, динамика по месяцам, финансовые показатели. |
| Архив СЭМД (`06_semd_archive.json`)     | Поддержка                  | Что произошло с конкретным документом — поиск по любому идентификатору, статус, ошибки, цепочка сообщений. |
| Клиентский сервис (`07_client_service.json`) | Клиент                | Состояние обмена в разрезе одной клиники.                                              |
| Клиентская BI-аналитика (`08_client_bianalytic.json`) | Клиент             | Аналитический срез обмена для предоставления клиенту.                                  |

Дашборды хранятся как код в `metabase_dashboards/` и импортируются при старте контейнера скриптом `metabase/setup-dashboards.sh`.

Карточки строятся поверх аналитических представлений (`v_egisz_documents_enriched_ui`, `v_rpt_*_ui`), а не поверх сырого журнала. Парсинг и классификация выполнены в DWH заранее, поэтому Metabase обращается к готовым колонкам, а изменение логики разбора или классификации применяется правкой SQL-функции и обновлением витрины — без переработки самих дашбордов.

Фильтры по клинике (`JID`), типу СЭМД и статусу документа задаются декларативно в `field_filter_defaults.yaml`; список доступных значений подставляется из данных автоматически.

## Эксплуатация

**Запуск и остановка.** `up.ps1` управляет Airflow и Metabase в Docker Desktop Kubernetes через параметр `-Action`:

```
.\up.ps1                          # = -Action Start: полный запуск/обновление Airflow + Metabase
.\up.ps1 -Action Airflow          # только Airflow
.\up.ps1 -Action Metabase         # только Metabase
.\up.ps1 -Action Stop             # полная остановка
.\up.ps1 -Action Stop-Airflow     # остановить только Airflow
.\up.ps1 -Action Stop-Metabase    # остановить только Metabase
```

Docker используется только для сборки локальных образов `egisz-airflow-worker` и `egisz-metabase`; runtime остаётся Kubernetes (Airflow через Helm, Metabase через `kubectl`).

**DWH-схема** создаётся отдельной командой против целевой БД (см. [DWH-модель](#dwh-модель)). Перед первым прогоном нужны базовые объекты:

```
CREATE ROLE egisz LOGIN PASSWORD 'egisz';
CREATE DATABASE dwh_egisz OWNER postgres;
```

После `helm uninstall airflow` служебные PVC могут остаться в кластере; чтобы начать metadata DB с нуля, удалите PVC Airflow PostgreSQL отдельно (`kubectl get pvc` → `kubectl delete pvc <name>`).

**Мониторинг здоровья.** Основная точка — дашборд `02_service.json`. Смотреть в первую очередь: время последнего `update_watermark` (lag DAG — в пределах нескольких минут), значение `last_logid` в `elt_state` (должно расти), долю ошибочных DAG-runs в Airflow UI. Сигнал `data_freshness` в `v_health_signals_ui` переходит в красный, если последний документ старше 24 ч.

**Типовые сценарии «жалоба → куда смотреть»:**

- «Клиника X сегодня не отправляет документы» → `01_operational.json`, фильтр по клинике, динамика за неделю; при недавнем провале проверить `02_service.json` — не остановился ли ETL.
- «Документ не дошёл до РЭМД, дайте детали» → `06_semd_archive.json`, поиск по `localUid` или `messageId`: статус, ошибки, цепочка сообщений.
- «Почему в этом месяце выросла доля отказов» → `04_quality_and_errors.json`, топ `error_type` за месяц.
- «Сводка для руководителя» → `05_executive.json`.
- «Вырос `Неизвестная ошибка`» → `04_quality_and_errors.json`, отфильтровать этот тип, посмотреть исходные сообщения, добавить правило в `db/parts/30_error_rules.sql`, прогнать `db/dwh_init.sql` и переклассифицировать историю.

**CI.** В GitHub Actions (`.github/workflows/ci.yml`) проверяются pytest-тесты из `tests/`, прогон `db/dwh_init.sql` против чистой PostgreSQL (с двойным запуском для проверки идемпотентности) и резолвер field-фильтров (повторный прогон — no-op). Падение любой проверки блокирует мерж.

## Приложение: полный список правил классификации

В seed-таблице `egisz_error_interpretation_rules` (`db/parts/30_error_rules.sql`) задано **86 правил**. Порядок ниже — по возрастанию `priority` (порядок срабатывания). Правило применяется, когда совпали `code` (если задан) **и** паттерн текста сообщения; `—` в колонке `code` означает «правило только по тексту», `—` в колонке паттерна — «правило только по коду». Регэкспы приведены без флага `(?is)` и могут быть усечены — источник истины в коде.

| № | Тип ошибки | code | Паттерн текста (regex) | Группа |
| --- | --- | --- | --- | --- |

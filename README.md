## Контекст: ЕГИСЗ, СЭМД, обмен и где в нём наш контур

**ЕГИСЗ** — Единая государственная информационная система в сфере здравоохранения. С точки зрения медицинской организации (МО) основное взаимодействие с ЕГИСЗ — это передача структурированных электронных медицинских документов (**СЭМД**) в подсистему **РЭМД** (Реестр электронных медицинских документов). Каждый СЭМД (выписной эпикриз, протокол осмотра, направление, заключение и так далее — десятки видов документов) должен быть зарегистрирован в РЭМД.

Обмен с РЭМД асинхронный по схеме «отправил — получил подтверждение позже». МИС клиники формирует CDA-документ, оборачивает его в SOAP-конверт, ставит ЭП, отправляет в РЭМД и получает синхронный технический ответ («принято к обработке»). Спустя секунды или часы приходит асинхронный callback с результатом: документ либо зарегистрирован и получил `emdrId`, либо отклонён со списком ошибок (Schematron, XSD, бизнес-правила, проблемы с подписью или со справочниками НСИ). Один логический «отправил СЭМД» — это, как правило, **минимум два сообщения** в журнале обмена.

Наша компания работает как **интегратор**: между МИС клиник и инфраструктурой ЕГИСЗ стоит проксирующий шлюз. Он принимает от МИС уже подписанный документ, маршрутизирует его в РЭМД, получает callback и возвращает его клинике. Все эти сообщения шлюз пишет в свою журнальную БД — **`proxy_egisz`** на Firebird 5. Эта БД — нормализованный лог обмена для нужд самого шлюза, она не приспособлена ни для аналитики, ни для эксплуатационных отчётов.

Поверх этого лога живёт наш сервис: каждые 5 минут мы инкрементально вычитываем новые строки журнала, парсим SOAP/XML на стороне PostgreSQL, склеиваем исходящее сообщение с его callback, нормализуем статусы и причины отказов, и формируем витрину фактов, на которой строятся дашборды Metabase.

## Что делает сервис и на какие вопросы отвечает

С практической точки зрения сервис закрывает пять задач:

**Контроль отправки.** Видеть, что МИС клиник в принципе отправляют СЭМД через шлюз: число сообщений в час и в день, разрезы по клинике и типу документа, провалы и всплески. Если клиника прекратила отправку — это часто означает, что у неё что-то сломалось в МИС, и узнаём об этом по нашим графикам раньше, чем по тикету.

**Контроль обратной связи.** Видеть, что на каждое исходящее СЭМД пришёл callback от РЭМД, и что callback корректно склеился с оригиналом по идентификаторам (`messageId`, `relatesTo`, `localUid`, `emdrId`). Документ, у которого callback не пришёл за разумное окно (обычно >24 ч) — кандидат на эскалацию.

**Контроль ошибок.** Видеть, какие СЭМД отвергнуты, и понимать причину в человекочитаемом виде. РЭМД возвращает ошибки длинными XML-конвертами с десятками `<item>` внутри, часть из них вообще не имеет внятного человеческого текста (только код `RUNTIME_ERROR` и т. п.). Сервис парсит этот XML и приводит каждую ошибку к одной из ~70 канонических категорий (СНИЛС пациента, специальность врача не в НСИ, истёк сертификат организации, сетевой таймаут, дубликат, и так далее).

**Контроль полноты.** Видеть, нет ли пропусков в обработке: документы без `emdrId`, документы со «странным» статусом, расхождения между журнальной таблицей и таблицей callback'ов, пустые поля там, где их быть не должно.

**Контроль самого сервиса.** Видеть, что наш ETL-контур жив: cursor двигается, mat-view'ы свежие, lag DAG в пределах нормы, нет накопившихся ошибок ETL. Под это есть отдельный пульт (`02_service.json`).

## Архитектура и поток данных

Сервис устроен по схеме **ELT**: извлечение и загрузка — на Python внутри Airflow, **все** трансформации — на стороне PostgreSQL внутри PL/pgSQL-функций. Этот выбор сделан осознанно: вся бизнес-логика парсинга XML, нормализации статусов, классификации ошибок и сборки фактовой таблицы живёт в одном месте — `db/parts/`, — и её можно прогонять, переигрывать и тестировать прямо в БД без поднятия Airflow. Python-слой при этом тонкий и не содержит знаний предметной области.

Компоненты и их ответственность:

| Компонент | База / технология | Что делает |
|---|---|---|
| Источник | Firebird 5, БД `proxy_egisz` | Журнал обмена шлюза. Не правим, только читаем. |
| Оркестратор | Apache Airflow 2.11.2, DAG `egisz_elt_dag` | Каждые 5 минут забирает дельту, грузит в DWH, запускает SQL-трансформацию, обновляет mat-view'ы, двигает курсор. |
| DWH | PostgreSQL, БД `dwh_egisz` | Raw-слой, справочники, фактовая таблица, ~17 PL/pgSQL-функций, 2 материализованных и 11 обычных view. |
| Визуализация | Metabase v0.60.2.5 | 6 дашбордов, ~75 native-карточек поверх `v_egisz_transactions_enriched_ui` и `v_rpt_*_ui`. |
| Деплой | Kubernetes (Docker Desktop по умолчанию) | Helm-чарт Airflow + ручной манифест Metabase, общий bootstrap-скрипт `up.ps1`. |

Сетевые порты для локального dev-стенда: Airflow на `localhost:8080`, Metabase на `localhost:3000`. Airflow держит свою служебную базу `airflow_db` (внутри Helm-чарта), Metabase — свою `metabase_app`. Аналитический DWH `dwh_egisz` живёт отдельно и не используется ни для каких служебных нужд.

Подключения хранятся в Airflow Connections, имена фиксированы: `proxy_egisz_fb` для Firebird-источника и `dwh_egisz_pg` для DWH. В DAG никогда не используется `os.getenv` или `.env` — только `BaseHook.get_connection`. Это позволяет переподключать инстансы (например, переключить чтение на read-replica) без правок кода.

## Источник: прокси-БД интегратора (Firebird `proxy_egisz`)

Из десятков таблиц шлюза мы читаем четыре. Логика отбора всегда инкрементальная по монотонно растущему ключу — Firebird поддерживает keyset-пагинацию через `WHERE col > :last_seen ORDER BY col ROWS :limit`, а `LIMIT/OFFSET` на этом диалекте использовать нельзя.

**`EXCHANGELOG`** — основной журнал. Каждая строка — это одно сообщение между шлюзом и РЭМД (как исходящее, так и входящее), с полями `LOGID` (PK, монотонный bigint), `LOGDATE`, `CREATEDATE`, `MSGID`, `LOGSTATE` (числовой код состояния), `LOGTEXT` (короткий текст лога) и `MSGTEXT` (полный SOAP/XML payload как BLOB). Именно `MSGTEXT` — основной материал для парсинга: из него извлекаются идентификаторы документа, код и название СЭМД, статус, ошибки.

**`EGISZ_MESSAGES`** — отдельная таблица сообщений с короткими метаданными. Поля `EGMID` (PK, bigint), `MSGID`, `REPLY_TO` (идентификатор сообщения, на которое это callback), `DOCUMENT_ID` (`emdrId` или `localUid`), `CREATED_AT`. Без полного XML — он есть только в `EXCHANGELOG.MSGTEXT`. Используется для построения связей «исходящее ↔ callback», когда XML исходного и ответного сообщений лежат в разных строках `EXCHANGELOG`.

**`JPERSONS`** и **`EGISZ_LICENSES`** — справочные таблицы. Из первой берём `JID` (внутренний ID организации в системе шлюза), `JNAME`, `JINN`, `JADDR` и складываем в `dim_organizations`. Из второй — `mo_uid` (OID организации в ФРМО), привязку к `JID`, доменное имя домейна МИС, даты действия лицензии; складываем в `dim_licenses`. Эти два справочника позволяют по любому идентификатору в callback (`mo_uid` или GOST-хост) дотянуться до человекочитаемого имени клиники и наоборот.

Идентификаторы в обмене с РЭМД часто приходят обёрнутыми (`<urn:uuid:...>`), и одно и то же сообщение может фигурировать в трёх формах: голый UUID, `urn:uuid:UUID`, `<urn:uuid:UUID>`. На уровне БД для них работает `egisz_normalize_message_id()`, на эту нормализованную форму повешены функциональные индексы — поэтому JOIN-ы между журналом и таблицей сообщений быстрые независимо от того, в каком виде идентификатор лежит в источнике.

## ELT-конвейер Airflow

DAG `egisz_elt_dag` собран на TaskFlow API, запускается каждые 5 минут (`*/5 * * * *`), `max_active_runs=1` (параллельные прогоны запрещены, чтобы не было гонки за watermark). Размер батча `BATCH_SIZE = 5000` — это компромисс между задержкой подъёма данных и нагрузкой на Firebird; на ровном потоке батча хватает, на пиках DAG идёт в догон несколько прогонов подряд.

Последовательность задач:

```
sync_dimensions
  ▸ extract_from_proxy
    ▸ load_to_dwh
      ▸ analyze_raw_tables
        ▸ transform_data
          ▸ refresh_materialized_views
            ▸ update_watermark
```

**`sync_dimensions`** делает полную перезагрузку справочников `dim_organizations` и `dim_licenses` из Firebird. Это дёшево (тысячи строк), и UPSERT через `ON CONFLICT (jid)`/`ON CONFLICT (id)` гарантирует, что новые клиники и новые лицензии видны сразу в первом же прогоне после изменения. Задача поставлена в начало DAG, чтобы остальные шаги уже работали с актуальными справочниками.

**`extract_from_proxy`** читает текущий watermark из таблицы `elt_state` (`last_log_id`, `last_egmid` для пайплайна `egisz`), берёт батч новых `EXCHANGELOG` по `LOGID > last_log_id` и батч новых `EGISZ_MESSAGES` по `EGMID > last_egmid`. После этого дополнительно подтягивает связанные старые сообщения, на которые ссылаются свежие строки журнала (через `<messageId>`, `<relatesToMessage>`, `<relatesTo>`, `<localUid>`, `<DOCUMENTID>` в XML и через `MSGID` в журнальной строке) — это позволяет в одной транзакции собрать всю цепочку «исходящее + callback», даже если callback пришёл много позже исходного сообщения. Результат — словарь с двумя списками строк (`rows` для `exchangelog_raw`, `message_rows` для `egisz_messages_raw`) плюс служебные счётчики и максимальные ID для следующего шага.

**`load_to_dwh`** делает UPSERT в `exchangelog_raw` и `egisz_messages_raw` через `INSERT ... ON CONFLICT DO UPDATE` (PK — `logid` и `egmid` соответственно). Идемпотентность гарантирована: повторный прогон того же батча не приведёт к дублям, повторное появление более свежей версии той же строки (например, через ретрай в шлюзе) перетрёт старую.

**`analyze_raw_tables`** запускает `ANALYZE public.exchangelog_raw` и `ANALYZE public.egisz_messages_raw` в autocommit-сессии — только для тех таблиц, в которые в этом батче что-то загружалось. Эта задача добавлена после инцидента: до неё PostgreSQL planner после первичного bulk-COPY использовал `pg_class.reltuples = 0` и выбирал seq-scan по `exchangelog_raw` (~1.2 ГБ) для каждой строки запроса, обходя имеющиеся функциональные индексы `msgid_norm` / `document_id_norm`. Карточки Metabase с очередью «документы без ответа» зависали на 8-16 минут, забивая весь пул соединений. Autovacuum на спокойном пайплайне не успевал запустить ANALYZE достаточно часто. После явного ANALYZE на каждом батче (стоит ~1 с — это sample-scan) проблема ушла. Эта задача — обязательная и в DAG, и в контракте AGENTS.md §2.

**`transform_data`** вызывает единственную SQL-функцию `public.egisz_transform_raw_to_facts(min_log_id, max_log_id, min_egmid, max_egmid)`. Окно курсора в параметрах — для того, чтобы функция работала только с новыми строками, а не пересчитывала всю историю. Внутри функции — парсинг `MSGTEXT`, нормализация статусов, классификация ошибок, склейка с callback, UPSERT в `fact_egisz_transactions`. Кандидатами для обработки идут не только свежие строки журнала, но и старые `exchangelog_raw`, к которым в текущем EGMID-окне прилетел поздний callback из `egisz_messages_raw`. Вся бизнес-логика трансформации живёт в этой функции и в её хелперах — в Python ничего этого нет.

**`refresh_materialized_views`** обновляет две тяжёлые витрины — `v_egisz_transactions_enriched_ui` и `v_stg_channel_errors_by_document` — через `REFRESH MATERIALIZED VIEW CONCURRENTLY`. CONCURRENTLY — чтобы не блокировать SELECT'ы от Metabase, которые могут идти одновременно (это требует уникального индекса на mat-view, он есть на `transaction_id`). 11 обычных view (`v_rpt_*_ui`, `v_health_*_ui`) — view, а не mat-view, и пересчитываются на каждый SELECT поверх свежей mat-view; нагрузка такого пересчёта на типовых запросах копеечная.

**`update_watermark`** UPSERT'ит `elt_state` через `GREATEST(current, new)`, чтобы случайный сбой в порядке прогонов или ручной запуск более старого батча не откатил cursor назад. Это последняя строка в DAG, и watermark двигается только после успеха всех предыдущих шагов — если что-то упало, на следующем прогоне те же данные будут перечитаны заново.

XCom-payload, который таски передают друг другу, имеет фиксированную форму:

```python
{
    "count": int,           # EXCHANGELOG-строк прочитано
    "message_count": int,   # EGISZ_MESSAGES-строк (включая «подтянутые» старые)
    "last_log_id": int,     # watermark на начало этого прогона
    "last_egmid": int,
    "max_id": int,          # максимальный LOGID в этом батче
    "max_egmid": int,       # максимальный EGMID в основном батче (без подтянутых)
    "rows": list[dict],     # сериализованные EXCHANGELOG-строки
    "message_rows": list[dict],
}
```

Эта форма зафиксирована в контракте: ровно эти ключи ожидают все задачи downstream, добавление/переименование требует синхронной правки и в `airflow/dags/egisz_elt_dag.py`, и в `AGENTS.md`.

## Парсинг и нормализация payload

Вся работа с XML делается на стороне PostgreSQL через 6 утилитных функций в `db/parts/20_functions_parsing.sql`. Python даже не пытается парсить SOAP/XML на стороне DAG — это упрощает и DAG, и тестирование (правила парсинга проверяются прямыми SELECT'ами в БД).

**`egisz_xml_text(payload text, tag_name text) RETURNS text[]`** — извлекает все вхождения текста между тегами вида `<*:tag_name>...</*:tag_name>`, агностично к префиксу неймспейса. Реализована как `regexp_matches` с безопасной экранировкой имени тега (выкидываем всё, кроме `[A-Za-z0-9_:-]`). Это та функция, через которую из `MSGTEXT` достаются `messageId`, `relatesToMessage`, `relatesTo`, `localUid`, `DOCUMENTID`, статус, эндпоинт обратного вызова и десятки других полей.

**`egisz_normalize_message_id(value text) RETURNS text`** — приводит идентификаторы вида `<urn:uuid:dd73fc79-e2e6-479c-a285-2a470fc4f04e>`, `urn:uuid:dd73fc79-...`, `dd73fc79-...` к одной канонической форме (голый UUID). На эту нормализованную форму повешены функциональные индексы — `idx_egisz_messages_msgid_norm`, `idx_fact_egisz_message_id_norm`, `idx_fact_egisz_relates_to_norm` — чтобы JOIN-ы по идентификаторам не делали seq-scan.

**`safe_cast_timestamptz(p_text text) RETURNS timestamptz`** — безопасный каст текста в `timestamptz`, возвращает `NULL` вместо исключения. Используется потому, что в payload из источника поля даты иногда приходят в нескольких форматах одновременно (с/без таймзоны, ISO/локальный), и одна битая строка не должна валить весь батч.

**`egisz_clean_host(p_text text)`** и **`egisz_extract_jid_from_endpoint(p_text text)`** — пара функций для разбора эндпоинта callback. В callback от РЭМД целевой URL клиники приходит с GOST-хостом вида `gost-1234.infoclinica.lan`, из которого `1234` — это `JID` организации. Эти функции вытаскивают идентификатор клиники из URL и нормализуют сам хост, чтобы по нему можно было JOIN-ить со справочником.

**`egisz_clean_text_value(p_text text)`** — общая нормализация любого текстового поля: схлопывание whitespace, удаление BOM/неразрывных пробелов, обрезка. Используется при подготовке полей для UI, чтобы в Metabase не появлялись «дубликаты», отличающиеся только пробельным шумом.

**`egisz_normalize_semd_code(p_text text)`** — нормализация кода СЭМД (бывает с префиксами/суффиксами от разных версий шаблонов).

Извлечение и парсинг payload в `egisz_transform_raw_to_facts` идут в три прохода: сначала достаются все ключевые поля идентификаторов и статуса, затем парсятся ошибки (если они есть в ответе РЭМД), затем склеивается JSON-payload ошибок для дальнейшего drill-down в Metabase.

## Классификация ошибок

РЭМД возвращает ошибки несколькими способами: машиночитаемым кодом в SOAP-faultcode (`VALIDATION_ERROR`, `RUNTIME_ERROR`, `ASYNC_RESPONSE_TIMEOUT`, ...), плюс текстом, плюс отдельными `<item>` внутри ответного XML (часто десяток в одном callback), плюс инлайн-Schematron-кусками в человеческом тексте. Если просто складывать сырой `error_message` — получится несколько тысяч уникальных формулировок, которые невозможно агрегировать в дашборде.

Поэтому каждая ошибка проходит через классификатор и приводится к одной из **69 канонических категорий** плюс `Неизвестная ошибка`. Классификация декларативная: правила лежат в таблице `egisz_error_interpretation_rules` (`db/parts/30_error_rules.sql`) и применяются функцией `egisz_error_interpretation_type(error_code text, error_message text)` (`db/parts/40_functions_errors.sql`). Каждое правило — это тройка `(match_code, match_pattern, interpretation)` с приоритетом: «если SOAP-faultcode равен такому-то И/ИЛИ текст матчится по такому-то regex — отнеси ошибку к такому-то типу».

Внутри одного callback РЭМД может прислать несколько `<item>` разных типов; в `error_type` они дедуплицируются и склеиваются через ` · `. Если ни одно правило не сработало — `error_type` получает значение `Неизвестная ошибка`; такие случаи видны в дашборде `04_quality_and_errors.json` и служат сигналом, что в правила нужно добавить новую формулировку.

Содержательные группы (для дашбордов), на которые в итоге распадаются 69 правил:

| Группа | Описание |
| --- | --- |
| **Структурная валидация СЭМД** | Schematron, XSD, парсинг XML. Чинится на стороне МИС или шаблонизатора;  |
| **Справочники НСИ** | Устаревшая версия справочника, отсутствующий код, неверное наименование. |
| **Регистры пациента и медработника** | ФРМР, ФРМО, ГИП: несовпадение СНИЛС, должности, отчества, подразделения. |
| **Организация и лицензии** | ОГРН в СЭМД не совпадает с ФРМО, лицензия не найдена, организация не привязана к РМИС. |
| **Регистрация ИС в РЭМД** | `ACCESS_DENIED`, `DISABLED_RMIS`, `NO_RMIS`. |
| **Электронная подпись и сертификаты** | Сертификат истёк, отозван, недействителен на дату документа или недоступен CRL/OCSP. |
| **Документооборот РЭМД** | Документ уже зарегистрирован, не найден, запрос дублируется или аннулирован. |
| **Получение файла ЭМД из ИС** | `getDocumentFile` упал, файл не передан, МИС недоступна. Это обратное направление: РЭМД хочет получить файл по запросу и не может. |
| **Технические ошибки РЭМД** | `INTERNAL_ERROR`, `RUNTIME_ERROR`, `ASYNC_RESPONSE_TIMEOUT`, недоступность УЦ. Обычно решаются повторной отправкой. |
| **Сетевая ошибка** | Таймауты, обрывы, проблемы между нашим шлюзом и РЭМД. |
| **Неизвестная ошибка** | Final fallback. Должен оставаться в районе единиц процентов; если этот тип растёт, нужно добавлять правила. |

Полная таблица всех 69 правил (с конкретными `code` и regex'ами) — в [приложении ниже](#полный-список-правил-классификации) и в коде в `db/parts/30_error_rules.sql`.

Канонический результат сохраняется в фактовой таблице в трёх полях:

- `error_type` — короткий канонический тип для группировки в дашбордах.
- `error_summary` — короткая пользовательская интерпретация (для отображения в карточках).
- `error_json_text` — полный JSON со всеми `<item>`-ами, отнормализованными к структуре `{code, message, type}`. Это поле для drill-down: открыл документ в `06_semd_archive.json` — видишь весь набор причин.

Отдельная view `v_rpt_error_interpretations_ui` раскрывает интерпретации построчно — для аудита правил и поиска неклассифицированных шаблонов.

## DWH-модель

Схема живёт целиком в файле `db/dwh_init.sql` (тонкий собиратель через `\i`) и 11 модулях под `db/parts/`. Никаких миграционных файлов нет: новый dev-DWH полностью разворачивается одним прогоном `psql -f db/dwh_init.sql`, любые изменения схемы добавляются прямо в нужный модуль через идемпотентные паттерны (`CREATE TABLE IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, `ALTER TABLE ... IF EXISTS`, `INSERT ... ON CONFLICT`). Скрипт можно безопасно перезапускать сколько угодно раз — он не пересоздаёт данных и не теряет уже существующие.

Слои DWH:

**Raw-слой** — `exchangelog_raw` (PK `logid`), `egisz_messages_raw` (PK `egmid`). Хранят полные строки журнала «как пришло из Firebird», без какой-либо трансформации; `MSGTEXT` лежит как `text`, без попытки парсить заранее. Сюда же относится таблица `elt_state` с watermark-курсорами по пайплайну.

**Справочники** — `dim_organizations` (PK `jid`: имя клиники, ИНН, адрес), `dim_licenses` (PK `id`: привязка `jid` ↔ `mo_uid`, домен МИС, срок действия), `dim_semd_types` (PK `code`: код, название, группа документа). `dim_semd_types` seedится прямо из `db/parts/10_tables.sql`.

**Фактовая таблица** — `fact_egisz_transactions` (PK `exchangelog_log_id`, FK на `exchangelog_raw`). Колонки: идентификаторы документа (`message_id`, `relates_to_id`, `local_uid_semd`, `emdr_id`, `doc_number`), временные метки (`log_date`, `creation_date`, `processed_at`), статус и ошибки (`status`, `error_message`, `error_code`, плюс enriched-поля), организационные признаки (`jid`, `org_oid`, `egmid`), классификация документа (`semd_code`, `semd_name`), callback-url. Одна строка фактовой таблицы = одна логическая транзакция «отправка СЭМД и её результат», независимо от того, сколько физических сообщений было в журнале.

**Materialized view-слой** — две тяжёлые витрины:

- `v_egisz_transactions_enriched_ui` — основная аналитическая витрина, JOIN-ит факты со всеми справочниками, добавляет человекочитаемые названия колонок (русскоязычные алиасы для Metabase), нормализует статусы, добавляет день/неделю/месяц-агрегаты. Под неё повешены индексы на `transaction_id`, `День`, `JID клиники` — Metabase читает её как обычную таблицу.
- `v_stg_channel_errors_by_document` — агрегат по ошибкам канала на уровне документа. Используется как для дашборда «Ошибки канала», так и для подзапросов в других view.

**View-слой отчётов** (`v_rpt_*_ui`) 

`v_rpt_network_errors_detail_ui`, `v_rpt_documents_no_response_ui`, `v_rpt_semd_archive_ui`, `v_rpt_clinic_connectivity_daily_ui`, `v_rpt_connectivity_global_daily_ui`, `v_rpt_error_interpretations_ui`, `v_rpt_semd_archive_ui`

**Health-view** (`v_health_*_ui`) — служебные view для дашборда `02_service.json`: `v_health_by_clinic_ui` (доля успеха по клиникам), `v_health_proxy_db_ui` (свежесть данных в Firebird), `v_health_signals_ui` (сводные сигналы по контуру). Эти view не зависят от факта классификации — они показывают «жив ли сам сервис», а не «есть ли проблемы у клиники».

Порядок модулей в `db/parts/` отражает порядок зависимостей и обязателен:

| Модуль | Назначение |
|---|---|
| `00_bootstrap.sql` | Роль `egisz` и базовые GRANT'ы на schema `public`. |
| `10_tables.sql` | Таблицы, dim_semd_types seed, индексы (включая функциональные). |
| `20_functions_parsing.sql` | Утилиты парсинга XML и нормализации идентификаторов. |
| `30_error_rules.sql` | Таблица `egisz_error_interpretation_rules` и seed правил. |
| `40_functions_errors.sql` | Функции классификации, сборки JSON ошибок, нормализации лейблов СЭМД. |
| `50_transform.sql` | Главная функция `egisz_transform_raw_to_facts`. |
| `60_drop_dependents.sql` | Дроп view'ов и legacy-колонок перед пересборкой (нужно для смены типов). |
| `70_views_core.sql` | `v_egisz_transactions_enriched_ui` (MV) и `v_rpt_error_interpretations_ui`. |
| `75_views_stg.sql` | `v_stg_channel_errors_by_document` (MV) и `v_stg_channel_network_errors_by_document` (alias view). |
| `80_views_rpt.sql` | Все `v_rpt_*_ui` (network, no_response, semd_archive, connectivity). |
| `90_views_health_and_finalize.sql` | `v_health_*_ui` view, проверка GRANT'ов, `REFRESH MATERIALIZED VIEW`. |

Все объекты в схеме принадлежат роли `egisz`. Metabase подключается под отдельной read-only ролью `postgres` (имя унаследовано исторически — для read-only доступа).

## Аналитика и дашборды Metabase

В сервисе шесть дашбордов, всего ~75 native-карточек. Дашборды импортируются в Metabase при старте контейнера скриптом `metabase/setup-dashboards.sh`; JSON-определения лежат в `metabase_dashboards/`. Field-фильтры для дашбордов (то, что в Metabase позволяет фильтровать по «JID клиники» и видеть выпадающий список реальных значений из БД) настраиваются скриптом `scripts/apply_metabase_field_filters.py` по декларативным правилам из `metabase_dashboards/field_filter_defaults.yaml`.

Назначение каждого дашборда:

| № Название · файл | Назначение | Источники данных | Срезы |
|---|---|---|---|
| **01 Оперативный мониторинг** `01_operational.json` · 12 карточек | Контроль текущего состояния потока отправки. Счётчики за час и день, динамика доли ошибок, топ типов ошибок за сутки. | `v_egisz_transactions_enriched_ui` | Клиника (`JID`), тип СЭМД, тип ошибки (`error_type`), час, день |
| **02 Сервис и healthcheck** `02_service.json` · 16 карточек | Контроль состояния ETL-контура и канала шлюза. Lag последнего `update_watermark`, доля сбоев, объёмы по клиникам. | `v_health_by_clinic_ui` `v_health_proxy_db_ui` `v_health_signals_ui` `v_egisz_transactions_enriched_ui` | Клиника, день (heatmap), сигнал деградации канала |
| **03 Документы без ответа** `03_documents_no_response.json` · 5 карточек | Рабочая очередь эскалации. Список СЭМД, по которым callback не получен за настраиваемое окно (норматив >24 ч). | `v_rpt_documents_no_response_ui` | Клиника, тип СЭМД, возраст ожидания (ч) |
| **04 Ошибки и качество данных** `04_quality_and_errors.json` · 25 карточек | Детальный анализ отказов. Успешность отправки, топ `error_type` по 69-категорийной классификации, доля неклассифицированных ошибок, целостность идентификаторов. | `v_egisz_transactions_enriched_ui` `v_stg_channel_errors_by_document` `v_rpt_error_interpretations_ui` | Клиника × тип СЭМД (heatmap), `JID`/OID, период |
| **05 Сводная статистика** `05_executive.json` · 15 карточек | Управленческий срез без технических деталей. Общая доля успеха, топ причин отказов в процентах, динамика месяц к месяцу. | `v_egisz_transactions_enriched_ui` `v_rpt_connectivity_global_daily_ui` | Месяц, тип СЭМД, клиника |
| **06 Архив СЭМД** `06_semd_archive.json` · 2 карточки | Поиск конкретного документа по любому известному идентификатору. Полная цепочка сообщений: статус, ошибки, callback-URL, временны́е метки. | `v_rpt_semd_archive_ui` | `localUid`, `emdrId`, `messageId`, `LOGID`, `EGMID` |

Большинство карточек работает поверх `v_egisz_transactions_enriched_ui` (основная mat-view) и `v_rpt_*_ui` (специализированные view). Это даёт два важных свойства: во-первых, Metabase не парсит JSON/XML на своей стороне — все нужные поля уже разложены в колонки; во-вторых, при изменении логики парсинга или классификации достаточно поправить SQL-функцию и `REFRESH MATERIALIZED VIEW` — дашборды подхватят изменения без правок JSON.

Field-фильтры (`metabase-field-filters` в JSON-карточках) важны для UX: они позволяют пользователю выбирать «JID клиники» из выпадающего списка реальных значений, а не вводить руками. Маппинг template-tag → (`table_ref`, `field_name`) задаётся в `metabase_dashboards/field_filter_defaults.yaml` в декларативном виде с приоритетами и условиями (`query_contains`, `display_name_contains`, и т. д.). Резолвер `scripts/apply_metabase_field_filters.py` применяется к JSON-карточкам автоматически при старте Metabase и при правке дашбордов; повторный прогон — no-op.

## Эксплуатация

**Локальный запуск и остановка.** `up.ps1` управляет Airflow и Metabase в Docker Desktop Kubernetes:

```powershell
.\up.ps1                       # полный запуск/обновление Airflow + Metabase
.\up.ps1 -Action Stop          # полная остановка без удаления PVC и настроек Metabase
.\up.ps1 -Action Airflow       # только запуск/обновление Airflow
.\up.ps1 -Action Metabase      # только запуск/обновление Metabase
.\up.ps1 -Action Stop-Airflow  # только остановка Airflow
.\up.ps1 -Action Stop-Metabase # только остановка Metabase
```

DWH-схема создаётся отдельно, разовой командой против целевой БД:

```bash
psql -U postgres -d dwh_egisz -v ON_ERROR_STOP=1 -f db/dwh_init.sql
```

Этот же файл повторно прогоняется при обновлении DWH-модели — он идемпотентный. Перед первым прогоном нужны базовые объекты на стороне `postgres`:

```sql
CREATE ROLE egisz LOGIN PASSWORD 'egisz';
CREATE DATABASE dwh_egisz OWNER postgres;
```

**Connections.** В Airflow должны быть заведены два connection с фиксированными именами: `proxy_egisz_fb` (Firebird, проксирующая БД) и `dwh_egisz_pg` (PostgreSQL, DWH). Реквизиты лежат в k8s-секретах: `k8s/airflow/airflow-connections-secret.yaml` (этот файл не коммитится; есть пример с суффиксом `.example.yaml`). Метabase-секреты — в `k8s/metabase/metabase-connections-secret.yaml` по той же схеме.

**Мониторинг здоровья.** Дашборд `02_service.json` — основная точка проверки. Что смотреть в первую очередь: время последнего `update_watermark` (lag DAG, должен быть в пределах нескольких минут), число записей в `elt_state` (должно быть >0 и расти), доля ошибочных DAG-runs в Airflow UI на странице DAG-а. Если что-то из этого деградирует — открыть Airflow UI и посмотреть, на каком шаге упал DAG.

**Типовые сценарии.** Дашборды покрывают почти всю эксплуатационную рутину; ниже краткий маппинг «жалоба → куда смотреть»:

- «Сегодня клиника X не отправляет документы» → `01_operational.json`, фильтр по клинике, посмотреть динамику за неделю; если провал свежий — проверить `02_service.json` (вдруг наш ETL встал).
- «Документ не дошёл до РЭМД, дайте детали» → `06_semd_archive.json`, поиск по `localUid` или `messageId`; видно статус, ошибки, цепочку сообщений.
- «Хочу понять, почему в этом месяце выросла доля отказов» → `04_quality_and_errors.json`, топ `error_type` за месяц с предыдущим, фильтрация по группам.
- «Раз в неделю нужна сводка для руководителя» → `05_executive.json`, выгрузка PNG/CSV.
- «У нас выросло число `Неизвестная ошибка`» → `04_quality_and_errors.json`, отфильтровать `error_type = 'Неизвестная ошибка'`, посмотреть raw-сообщения, добавить правило в `db/parts/30_error_rules.sql`, прогнать `psql -f db/dwh_init.sql`, переклассифицировать историю (это разовая batch-операция, обычно делается через SQL вручную).

**Troubleshooting производительности.** Если Metabase-карточки стали тормозить и подвисать в районе минут — первое, что проверить, это статистика planner'а:

```sql
SELECT relname, last_analyze, last_autoanalyze, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
WHERE relname IN ('exchangelog_raw', 'egisz_messages_raw', 'fact_egisz_transactions');
```

Если `last_analyze` пустой или старый — autovacuum не успевает, и DAG-задача `analyze_raw_tables` должна это компенсировать (а если не компенсирует — смотреть логи Airflow). Разовое лечение — `ANALYZE public.exchangelog_raw; ANALYZE public.egisz_messages_raw;` под суперпользователем.

**CI.** В GitHub Actions настроены три джобы (`.github/workflows/ci.yml`):
- `pytest` на тестах из `tests/`;
- прогон `db/dwh_init.sql` против чистой `postgres:16` (с двойным запуском для проверки идемпотентности);
- прогон `scripts/apply_metabase_field_filters.py` с проверкой «второй прогон — no-op».

Эти проверки крутятся на каждом push и PR. Падение любой из них — блокер мержа.

### Полный список правил классификации

Нумерация ниже = порядок чтения и документирования; в seed-таблице `egisz_error_interpretation_rules` правилам соответствуют `rule_code` + `priority`. Колонка «Правила определения» читается как «code равен X и текст совпадает с regex Y»; `ИЛИ` разделяет альтернативные способы попадания в ту же интерпретацию. Группировка по содержательным группам приведена в разделе [Классификация ошибок](#классификация-ошибок); источник истины — `db/parts/30_error_rules.sql`. При добавлении нового правила — расширять seed через `INSERT ... ON CONFLICT (rule_code) DO UPDATE`, отдельных миграционных файлов в проекте не предусмотрено.

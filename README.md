# EGISZ Airflow ELT

Сервис контроля интеграции между клиниками и ЕГИСЗ в компании-интеграторе. Решение собирает журнал обмена из исходной Firebird-системы, формирует аналитический DWH-слой в PostgreSQL и публикует отчёты Metabase для контроля отправки, обработки, ошибок и полноты обратных сообщений.

Проект используется как прототип целевого сервиса для последующей миграции в корпоративную инфраструктуру с отдельными Kubernetes-кластерами, базами данных и внутренними контурами сопровождения.

## Назначение сервиса

Сервис предназначен для эксплуатационного контроля интеграции между клиниками и ЕГИСЗ. Он отвечает на следующие вопросы:

- какие сообщения были отправлены в ЕГИСЗ;
- какие документы получили callback, а какие остались без ответа;
- какие ошибки вернул транспортный канал или РЭМД;
- по каким клиникам, типам СЭМД и типам документов возникают проблемы;
- насколько полны, связны и свежи данные интеграционного контура.

Ключевые цели сервиса:

| Цель | Что контролируется |
| --- | --- |
| Контроль отправки | факт появления исходящих сообщений в журнале обмена |
| Контроль обратной связи | наличие callback и корректное связывание ответа с исходным сообщением |
| Контроль ошибок | транспортные ошибки, ошибки валидации, ошибки бизнес-правил |
| Контроль полноты | документы без ответа, пропущенные связи, неполные данные |
| Контроль состояния сервиса | актуальность данных и стабильность интеграционного потока |

## Архитектура и роль компонентов

Сервис построен как ELT-контур: Python извлекает и загружает данные, а основная аналитическая логика выполняется в SQL.

### Основные компоненты

| Компонент | Назначение |
| --- | --- |
| Firebird `proxy_egisz` | источник журнала обмена и связанных данных |
| Airflow DAG | оркестрация инкрементальной выгрузки и обновления DWH |
| PostgreSQL `dwh_egisz` | raw-слой, факты, справочники, materialized views и отчётные витрины |
| Metabase | публикация эксплуатационных и управленческих отчётов |

### Принципы архитектуры

| Принцип | Описание |
| --- | --- |
| ELT | Python не реализует предметную аналитику, а передаёт данные в DWH |
| SQL-first | парсинг payload, нормализация статусов и интерпретация ошибок живут в PostgreSQL |
| Idempotency | загрузка и bootstrap безопасны для повторных запусков |
| Report-ready DWH | Metabase работает по готовым витринам, не парсит JSON/XML на своей стороне |
| Contract-driven | структура raw-слоя, фактов и витрин выступает как переносимый контракт для будущей миграции |

## Логика отбора и обработки данных

### Поток DAG

```text
sync_dimensions
  >> extract_from_proxy
  >> load_to_dwh
  >> transform_data
  >> refresh_materialized_views
  >> update_watermark
```

### Параметры инкрементального контура

| Параметр | Значение |
| --- | --- |
| `BATCH_SIZE` | `5000` |
| `schedule` | `*/5 * * * *` |
| `max_active_runs` | `1` |

### Этапы обработки

| Этап | Что делает | Результат |
| --- | --- | --- |
| `sync_dimensions` | синхронизирует справочники организаций и лицензий | актуальные `dim_organizations`, `dim_licenses` |
| `extract_from_proxy` | читает курсоры и забирает новые записи `EXCHANGELOG` и `EGISZ_MESSAGES` | батч сырых записей и связанной корреляции |
| `load_to_dwh` | выполняет UPSERT в raw-слой | обновлённые `exchangelog_raw`, `egisz_messages_raw` |
| `transform_data` | вызывает `public.egisz_transform_raw_to_facts(...)` | обновлённые факты аналитики |
| `refresh_materialized_views` | обновляет materialized views | актуальные витрины для Metabase |
| `update_watermark` | сохраняет `last_log_id`, `last_egmid` | продвинутые курсоры инкрементальной обработки |

### Логика отбора данных

| Источник | Ключ отбора | Назначение |
| --- | --- | --- |
| `EXCHANGELOG` | `LOGID` | основной журнал отправки и событий обмена |
| `EGISZ_MESSAGES` | `EGMID` | callback и связанные сообщения РЭМД/ЕГИСЗ |
| XML payload | `message_id`, `relates_to_id`, `local_uid_semd`, `emdr_id` | связывание исходящих и входящих сообщений, идентификация документа |

### Ключевые правила отбора

| Правило | Значение |
| --- | --- |
| Пагинация Firebird | keyset-пагинация через `WHERE col > :last_seen ORDER BY col ROWS :limit` |
| Продвижение курсора | только после успешной загрузки и трансформации |
| Повторная загрузка | через `INSERT ... ON CONFLICT DO UPDATE` |
| Связь сообщения и callback | через идентификаторы корреляции из payload, а не через Metabase |

## DWH-модель и SQL-слой

### Bootstrap схемы

Схема DWH создаётся единым bootstrap-скриптом:

```bash
psql -U postgres -d dwh_egisz -v ON_ERROR_STOP=1 -f db/dwh_init.sql
```

`db/dwh_init.sql` подключает модули `db/parts/` в фиксированном порядке.

### Модули DWH

| Модуль | Назначение |
| --- | --- |
| `db/parts/00_bootstrap.sql` | базовые роли и grants |
| `db/parts/10_tables.sql` | таблицы raw-слоя, справочники, факты, индексы |
| `db/parts/20_functions_parsing.sql` | функции парсинга XML/payload |
| `db/parts/30_error_rules.sql` | таблица и seed правил интерпретации ошибок |
| `db/parts/40_functions_errors.sql` | функции классификации и подготовки error-полей |
| `db/parts/50_transform.sql` | `egisz_transform_raw_to_facts` |
| `db/parts/60_drop_dependents.sql` | удаление зависимых объектов перед переопределением |
| `db/parts/70_views_core.sql` | основные пользовательские витрины |
| `db/parts/75_views_stg.sql` | staging/materialized views для ошибок канала |
| `db/parts/80_views_rpt.sql` | отчётные представления |
| `db/parts/90_views_health_and_finalize.sql` | health-витрины и финализация |

### Основные таблицы и витрины

| Объект | Роль |
| --- | --- |
| `elt_state` | хранение watermark-курсоров |
| `exchangelog_raw` | raw-слой журнала обмена |
| `egisz_messages_raw` | raw-слой callback и связанных сообщений |
| `dim_organizations` | справочник клиник/организаций |
| `dim_licenses` | справочник лицензий и связей `mo_uid` / `jid` |
| `dim_semd_types` | справочник типов СЭМД |
| `fact_egisz_transactions` | центральная таблица нормализованной аналитики |
| `v_egisz_transactions_enriched_ui` | основная витрина для Metabase |
| `v_stg_channel_errors_by_document` | витрина по ошибкам канала на уровне документов |

### Что хранится в фактах

| Группа полей | Содержание |
| --- | --- |
| Идентификаторы обмена | `message_id`, `relates_to_id`, `local_uid_semd`, `emdr_id`, `exchangelog_log_id` |
| Временные метки | дата отправки, дата обработки, дата создания документа |
| Статус | нормализованный итог обработки |
| Организационные признаки | клиника, JID, привязка лицензии, связанный `mo_uid` |
| Документ | код и наименование СЭМД, атрибуты документа |
| Ошибки | `error_type`, `error_summary`, `error_json_text`, технические детали |

### Логика интерпретации ошибок

| Шаг | Описание |
| --- | --- |
| Выделение ошибки | извлечение ошибки канала или XML-ошибки ответа |
| Нормализация | разбор `code`, `message`, технического текста |
| Классификация | применение правил интерпретации из SQL-слоя |
| Публикация в факты | запись канонизированных полей для отчётов и фильтрации |

## Metabase и отчётность

Metabase публикует отчёты по уже подготовленным витринам DWH.

### Основные дашборды

| Файл | Назначение |
| --- | --- |
| `01_operational.json` | оперативная динамика обмена |
| `02_service.json` | сервисные показатели и health |
| `03_documents_no_response.json` | документы без найденного ответа |
| `04_quality_and_errors.json` | ошибки и качество данных |
| `05_executive.json` | управленческая сводка по интеграции |
| `06_semd_archive.json` | архив СЭМД и связанных идентификаторов |

### Основные типы отчётов

| Тип отчёта | Что показывает |
| --- | --- |
| Оперативная динамика | объём отправки, обработки и изменения потока во времени |
| Документы без ответа | сообщения без callback и проблемные точки обработки |
| Ошибки канала | транспортные, сетевые и технические сбои |
| Ошибки документов | ошибки по СЭМД, клиникам, кодам и типам ошибок |
| Health-витрины | свежесть данных, сигналы деградации и полнота загрузки |
| Архивные выборки | ретроспективный анализ СЭМД и связанных сообщений |

### Скрипты Metabase

| Файл | Назначение |
| --- | --- |
| `metabase/setup-dashboards.sh` | provisioning коллекции, источника данных, карточек и дашбордов |
| `scripts/apply_metabase_field_filters.py` | настройка field filters для native query cards |
| `metabase_dashboards/field_filter_defaults.yaml` | декларативные правила field-filter mapping |

### Что делает provisioning Metabase

| Шаг | Действие |
| --- | --- |
| Регистрация источника | подключение DWH в Metabase |
| Проверка контракта | убеждение, что нужные витрины и поля существуют |
| Синхронизация схемы | актуализация metadata Metabase |
| Импорт карточек | создание native query cards |
| Импорт дашбордов | публикация пользовательских отчётов |

## Роль проекта как прототипа миграции

Этот репозиторий фиксирует предметную логику сервиса контроля интеграции, которую затем необходимо перенести в корпоративную среду.

### Что является переносимым контрактом

| Контракт | Что должно сохраниться при миграции |
| --- | --- |
| Raw-слой | структура `exchangelog_raw`, `egisz_messages_raw`, watermark-логика |
| SQL-трансформация | правила парсинга payload, построения фактов и интерпретации ошибок |
| Аналитические факты | структура `fact_egisz_transactions` |
| Отчётные витрины | пользовательские и health-представления |
| Metabase-отчёты | JSON-карточки, фильтры, смысл дашбордов |

### Что важно с точки зрения интегратора

| Направление | Значение |
| --- | --- |
| Контроль клиник | возможность видеть проблемные клиники и документы |
| Контроль канала | прозрачность транспортных и технических ошибок |
| Контроль РЭМД/ЕГИСЗ | отслеживание callback и задержек обратной связи |
| Контроль качества | единая аналитика по ошибкам и неполным данным |
| Переносимость | отделение предметной логики от локальной инфраструктуры |
